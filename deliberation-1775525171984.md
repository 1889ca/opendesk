### 1. Resolving the Audit Paradox: Cryptographic Shredding via Key-Per-Document

**The Better Mechanism:** We should instead implement **Cryptographic Shredding**. The audit log remains strictly append-only, but the *payload* of every sensitive operation is encrypted with a symmetric key specific to that document (or tenant/user). These keys are managed via an integrated Key Management Service (KMS). 
*   **How it solves the paradox:** When a GDPR deletion request is executed, OpenDesk destroys the specific KMS key. The Merkle tree's integrity remains perfectly intact (the ciphertext hashes still validate), but the data is mathematically unrecoverable, satisfying legal definitions of data destruction/anonymization in most jurisdictions.

**Hardest sub-problem:** Key rotation and key derivation hierarchy. If a document lives for 10 years, rotating its encryption key means re-encrypting the audit log, which violates the append-only rule. You need a robust key-wrapping architecture where the Document Encryption Key (DEK) is static but wrapped by a frequently rotated Key Encryption Key (KEK).

**Milestones:**

### 2. The Sovereign AI Permission Trap: RBAC-Enforced Vector Search

**Building on Claude’s Point 4 (Sovereign Document Intelligence):**
Local LLMs solve the *data residency* problem, but they introduce a massive *internal data leakage* problem. If you build RAG (Retrieval-Augmented Generation) over an organization's corpus, the LLM will inevitably retrieve snippets from documents the prompting user isn't allowed to see. **The Concrete Mechanism:** We cannot rely on post-filtering (fetching 100 relevant vectors and discarding 90 because of permissions—this destroys search recall). We must use **Vector Pre-filtering**. Since OpenDesk already uses PostgreSQL, we can leverage `pgvector`. Every vector embedding inserted into the DB must be tagged with a bitmask or array of the document's current Access Control List (ACL). 

**Hardest sub-problem:** Permissions in a collaborative suite are dynamic. If a folder's permission changes, propagating that ACL update to millions of vector metadata rows in real-time without locking the database is incredibly difficult. 

**Hypothesis:** We can abstract permission groups into integer-based materialized paths (e.g., `folder_id:group_id`), allowing the vector similarity search (`<=>`) to execute a fast index scan that strictly intersects with the user's active session roles *before* calculating cosine similarity.

Gemini's TEE proposal is directionally correct but the trust model described has a critical gap: **who verifies the attestation quote?** The sequence described—client requests hardware-signed quote, verifies code hash, releases KMS key—assumes the client has both the cryptographic tooling and the institutional knowledge to validate Intel TDX or AMD SEV-SNP attestation chains. Designing sovereign healthcare data infrastructure around hardware that has a track record of breaking in precisely the "trusted computation on untrusted host" scenario is a real risk that should be named, not elided.

**Near-term alternative with equivalent security properties:** Application-layer end-to-end encryption where the key never touches the server in plaintext — the same model Apple uses in iCloud Advanced Data Protection. PostgreSQL stores only ciphertext; Yjs operations encrypt at the client before transmission; the server orchestrates without reading. This works today, doesn't require TEE hardware procurement, and the trust model is straightforward enough that an auditor can explain it to a regulator without a cryptography degree. TEEs remain the right long-term answer for scenarios where you need the server to *compute* on plaintext (running AI inference on confidential documents) — that's the actual killer use case, not storage.

### The Gap Everyone Missed: Metadata Sovereignty

Every proposal in this conversation focuses on document *content*. But the communication graph—who is editing what with whom, when, at what frequency, from which devices—is often more sensitive than the content itself.

Consider: a health authority's document access logs reveal which physicians are working on which patients' cases, before any clinical note is written. A government ministry's co-editing patterns reveal which departments are collaborating on which policies, before any document is published. A First Nations band council's document timing patterns reveal which council members are driving which land claim strategies.

**Structural problem:** Every self-hosted deployment currently exposes this metadata to anyone with database access—which includes the MSSP sysadmin, the internal IT administrator, and any adversary who gains infrastructure access. Cloud providers are *worse* (they aggregate this across their entire customer base), but "slightly less bad than Google" is not the sovereign promise.

**Concrete mechanism:** **Metadata obfuscation via access pattern hiding.** At minimum: decoupling the real document ID from the observable access identifier (per-session pseudonymous document handles), bucketing edit timestamps to reduce precision, and separating the presence/websocket layer from the storage layer so the operator sees connection metadata but not which document is being edited. This is weaker than full Private Information Retrieval (PIR), which is computationally prohibitive at scale, but it's deployable and raises the attack cost substantially.

**Who this serves:** Specifically the legal profession (solicitor-client privilege extends to *the fact of consultation*, not just its content), journalists (source protection requires protecting the communication pattern, not just the message), and any organization where the metadata is itself the classified information.

**Milestones:**
1. Document handle pseudonymization — per-session ephemeral IDs that don't map to canonical document IDs in observable network traffic
2. Presence layer isolation — websocket presence server cannot read document content or canonical IDs
3. Operator access audit — what can an MSSP operator actually observe? Generate a formal threat model document
4. Differential privacy for analytics — usage statistics for tenants without individual-level attribution

### Challenge to the AGPL Developer Platform: The Government Exception Problem

The AGPL plugin ecosystem proposal has a structural problem that would prevent its adoption by exactly the organizations OpenDesk is targeting.

AGPL §13 requires that anyone who runs modified AGPL software over a network must publish the source. Government agencies that customize OpenDesk may have **statutory obligations preventing publication** — national security considerations, operational security for law enforcement, protection of indigenous community data schemas that themselves constitute proprietary cultural property under OCAP principles. A First Nations community building a custom plugin encoding their governance protocols cannot be legally compelled to publish it.

This isn't theoretical. It's why the US DoD has historically preferred BSD-licensed software over GPL-licensed alternatives for certain applications. It's why many government deployments of open-source software operate via legal ambiguity rather than clean compliance.

**The resolution isn't "weaken AGPL."** The resolution is a **two-tier model**:
- Core OpenDesk: AGPL — full copyleft, community property
- Plugin interface: LGPL or a purpose-specific "Sovereign Plugin Exception" modeled on the GCC Runtime Library Exception — you can build proprietary plugins against a defined API without those plugins triggering the AGPL copyleft

This is technically straightforward (define the plugin API surface as the LGPL boundary) but requires a deliberate licensing policy decision that should happen *before* the plugin ecosystem is built. Getting it wrong means either deterring government adoption or fragmenting the codebase into incompatible proprietary forks.

### New Segment: Temporal Sovereignty — Documents That Outlive Their Software

Every segment proposed assumes OpenDesk continues to exist as a functioning deployment. None addresses the scenario where it doesn't.

**Who faces this problem acutely:** Government archives (Canadian federal government has 30-year retention mandates for certain records), indigenous communities (treaty documents need to be verifiable by communities that don't yet exist), legal instruments (land titles, corporate bylaws), academic research data (50-year reproducibility horizon is normal).

**What it encompasses:** The architecture of document preservation independent of OpenDesk's operational continuity — spanning format migration, signature validity over time, and organizational discontinuity.

**Why cloud competitors can't solve this:** A document in Google Docs is accessible as long as Google exists and you maintain a Google account. That's not archival sovereignty — it's archival dependency with a plausible near-term horizon. Sovereign archival requires documents to be readable by software that doesn't exist yet, verified by organizations that haven't been formed yet, accessible to beneficiaries who don't know OpenDesk exists.

**Concrete mechanisms:**

*Format mortality:* Every document format becomes unreadable eventually. OpenDesk's export pipeline already supports multiple formats; a temporal sovereignty layer adds **migration chains** — automated conversion workflows that transform documents forward as formats evolve, preserving semantic fidelity rather than just byte-for-byte export. ODF is the right target format here; it's the one format controlled by a standards body rather than a single vendor.

*Signature expiry:* Digital signatures use certificates that expire. A document signed in 2026 will have an invalid signature in 2040 when the certificate chain is gone. The solution is **RFC 3161 trusted timestamping** at signing time — the timestamp authority countersigns the document hash before the certificate expires, creating a provably-valid-at-time-of-signing record that remains verifiable indefinitely. This is a solved standards problem; almost no document software implements it correctly.

*Software mortality:* The archival export must be interpretable without OpenDesk. This means **self-describing archives** — a ZIP containing the document in ODF, the signature chain, a human-readable metadata manifest, and enough schema documentation to interpret the structure without running OpenDesk. Think of it as a time capsule format.

**Milestones:**
1. RFC 3161 timestamping integration for all document signatures (deployable today, no new standards required)
2. Self-describing archival export format — the ZIP-as-time-capsule
3. Migration chain registry — formal declaration of migration paths between format versions
4. Long-term verification tooling — standalone CLI that verifies archival exports without requiring OpenDesk installation
5. Archival policy engine — organizations declare retention classes, each class gets automated migration triggers at configurable intervals

### The Thread Connecting All of This

There's an implicit assumption running through this entire conversation worth naming: **sovereignty is binary**. You either have it or you don't.

The more useful framing is **sovereignty as a spectrum with explicit, auditable levels** — and OpenDesk as the infrastructure that lets organizations know exactly where they sit. Every segment proposed maps to a different point on that spectrum: metadata hiding is harder than content encryption; temporal sovereignty is harder than operational sovereignty; air-gap is harder than MSSP-hosted.

The strategic implication: OpenDesk should publish a **Sovereignty Posture Framework** — an explicit, versioned, vendor-neutral standard defining what "Level 1 through Level 5" sovereign deployment means. This is a standards play, not just a product play. If OpenDesk defines the framework, every organization evaluating sovereign infrastructure evaluates against OpenDesk's criteria. The framework becomes a moat even for organizations that don't choose OpenDesk.

### Pushback on Claude's E2EE over TEEs: The Feature-Security Paradox
Claude rightly points out the usability and side-channel flaws of Trusted Execution Environments (TEEs) and suggests Application-Layer End-to-End Encryption (E2EE) as a safer alternative. If we mandate E2EE to neutralize the MSSP threat, the compute *must* move to the client edge. We compile lightweight embedding models (e.g., `all-MiniLM-L6-v2`) and RAG orchestration into WASM, running directly in the user's browser or a dedicated OpenDesk desktop client. ### Refining Temporal Sovereignty: ODF Destroys the Audit Chain
Claude's concept of Temporal Sovereignty (documents outliving their software) is vital, but relying on ODF (Open Document Format) as the archival artifact is a category error for a CRDT-based system. 

ODF is a snapshot of *state*. If you export to ODF, you destroy the entire Merkle-chained audit log of *history* (who edited what, when). In regulated environments (e.g., a 30-year treaty or complex litigation), the derivation of the text is often more legally significant than the final text itself.

**The Concrete Mechanism:** OpenDesk must pioneer a **Standardized CRDT Archive Format (SCAF)**. Instead of exporting the flat text, we export the entire Yjs operation DAG alongside the cryptographic signatures and VC proofs, packaged into a single SQLite file with a heavily documented, versionless read schema. ### New Segment: Adversarial Collaboration & Cryptographic Escrow (The Zero-Trust Deal Room)

**What it encompasses:** Workspaces designed not for trusted teams, but for mutually distrustful entities engaged in high-stakes negotiations (M&A, union bargaining, international treaties, sealed government procurement). It introduces **Threshold Cryptography** (e.g., Shamir's Secret Sharing) and **Time-Lock Encryption** into the document workflow.

**Who it serves:** Diplomatic corps, labor unions, corporate legal teams, and public procurement offices. 

**Why it's strategically important:** Current collaborative suites assume "Co-editing" equals "Trust." But a massive swath of sovereign workflows involves "Adversarial Collaboration." If a Canadian province and a First Nations band are negotiating a treaty on OpenDesk, neither wants the other (or the hosting MSSP) to unilaterally publish, leak, or lock the draft. 

**Concrete Mechanisms:**
1.  **Multi-Party Computation (MPC) Workspaces:** A workspace encrypted with a key that is split into $N$ shares. The document literally cannot be decrypted, exported, or submitted to the AI layer unless $M$ of $N$ stakeholders physically authenticate (e.g., 2 union reps and 2 management reps insert their smartcards). 
2.  **Cryptographic Time-Locks:** For procurement (RFP bidding) or embargoed press releases. Documents are encrypted natively within OpenDesk such that they mathematically cannot be decrypted until a specific block height on a public blockchain or a trusted time-beacon is reached. The MSSP cannot leak the RFP bids early, because the MSSP doesn't have the keys.
3.  **Blind Signatures:** Permitting "sealed" voting or approval workflows directly inside the document metadata without revealing *which* user cast which approval until the escrow conditions are met.

**Milestones:**
1. Implement a Threshold Cryptography key-sharing plugin over the KMS layer.
2. Build the "Escrow Workflow" state machine (defining M-of-N rules for document access/export).
3. Integrate an audited Time-Lock puzzle or trusted timestamp beacon for sealed document embargoes. 
4. UI for "Dual-Key" negotiation rooms, formally separating the UI concepts of "Internal Draft Space" and "Shared Negotiating Table."

By solving Adversarial Collaboration mathematically, OpenDesk captures the multi-billion dollar "Data Room" and legal negotiation market, which currently relies on archaic, non-collaborative PDF-shuffling precisely because real-time editing requires surrendering to a centralized cloud host.

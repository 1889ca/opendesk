# Decision #007: Phase 0.5 — Practical Erasure over CRDT Architecture Rewrite

**Date:** 2026-04-06
**Status:** Accepted
**Context:** The pillar sequencing deliberation (#005) proposed a 2-3 month "Phase 0.5" to modify Yjs internals for content-addressed blob storage, enabling cryptographic erasure via key destruction. This decision rejects that approach for now and opts for practical erasure instead.

## The Proposal (Rejected for Now)

Modify Yjs to store content hashes instead of inline values, with a content-addressable storage layer supporting per-blob encryption. Erasure = destroy encryption key = all blobs become unrecoverable.

**Why we're deferring:**
1. **No users yet.** The editor lacks comments, tables, images. Nobody is generating documents that need sub-document erasure.
2. **Yjs fork cost.** Modifying Yjs internals breaks TipTap, Hocuspocus, y-prosemirror, and cursor compatibility. Maintaining a fork is an indefinite tax.
3. **Practical erasure covers MVP needs.** Document deletion + CRDT compaction + snapshot-and-recreate physically destroys content without forking Yjs.
4. **Legal uncertainty.** No regulator has tested whether CRDT tombstones constitute "retained personal data" under GDPR. The legal question is unexplored.
5. **Opportunity cost.** 3 months on CRDT internals = 3 months not building editor features that determine whether anyone uses the product at all.

## The Decision: Practical Erasure (2 weeks)

Build erasure capabilities using existing primitives:

### 1. Full Purge Compaction
Enhance the existing compaction worker to support a "purge" mode:
- Materialize current document state as ProseMirror snapshot
- Destroy the entire Yjs binary state
- Rebuild a fresh Yjs doc from the snapshot
- All tombstones, operation history, and deleted content are physically gone

### 2. Document Deletion API
- Delete Yjs state from PostgreSQL
- Delete any S3 objects (cold storage)
- Invalidate Redis cache entries
- Remove permission grants
- Return confirmation with deletion timestamp

### 3. User Data Purge (GDPR SAR)
- Admin API to list all documents/grants/activity for a given user
- Bulk purge: delete all data associated with a user ID
- Generate a deletion receipt (timestamp, scope, operator)

### 4. Honest Documentation
Document erasure guarantees clearly:
- "Document content is physically destroyed after purge compaction"
- "CRDT operation metadata (timestamps, client IDs) may persist in database backups for the configured retention period"
- "This is equivalent to or better than erasure guarantees offered by major cloud office suites"

## Future: Content-Addressed Blobs (When Ready)

The content-addressed architecture IS the correct long-term answer. When the time comes:
- Yjs encoding plugin intercepts values during serialization
- Per-document symmetric keys, managed by KMS
- Erasure = key rotation + blob deletion
- Preserves Yjs sync protocol compatibility

**Trigger to revisit:** Enterprise customer with specific regulatory requirement for cryptographically verifiable sub-document erasure, OR legal clarity that CRDT tombstones = retained data under GDPR/PIPEDA.

## Trade-offs Accepted

| Aspect | Practical Erasure | Content-Addressed (Deferred) |
|--------|-------------------|------------------------------|
| Time to build | 2 weeks | 2-3 months |
| Yjs compatibility | Preserved | Broken (fork required) |
| Erasure granularity | Document-level | Sub-document / field-level |
| Cryptographic proof | No (physical deletion) | Yes (key destruction) |
| Offline client handling | No issue | Requires epoch rebasing |
| Regulatory adequacy | Sufficient for MVP | Required for regulated enterprise |

## Immediate Next Steps

1. Build Practical Erasure (2 weeks)
2. Invest saved time in Pillar 0: Editor Foundation (comments, tables, images)
3. Log the content-addressed architecture as a future option, not a current commitment

## Critical Findings

### C1 — Slides: image `src` accepts arbitrary schemes (XSS / DoS / SSRF)
**File:** `modules/app-slides/contract.ts:73`, `internal/render-image.ts:14`, `internal/parse-elements.ts:28`
**PR:** #490

`SlideElementSchema` declares `src: z.string().optional()` with no URL or scheme allowlist. `render-image.ts:14` assigns the value directly to `img.src`. A peer with write access (or any malicious migrated document) can write:

- `data:image/svg+xml;base64,<base64-encoded-svg-with-script>` — executes when rendered as `<object>`/`<iframe>` or used as CSS background; even within `<img>` it's a memory DoS (unbounded payload).
- `https://evil.com/beacon.png` — unauthenticated tracking pixel + SSRF surface against the renderer's network.
- `javascript:` (won't fire on `<img>` but reflects via copy-paste / export).

**Fix:** in `contract.ts`, replace with `z.string().max(2048).refine(u => /^https?:\/\/|^\/uploads\//.test(u))`. Reject `data:`, `javascript:`, `blob:`, `file:` in both `parse-elements.ts` and `yjs-element-insert.ts:insertElement`.

### C2 — Slides: rich-text HTML round-trips Yjs unsanitized (stored XSS)
**File:** `modules/app-slides/internal/render-text.ts:32`, `render-shape.ts:42`, `presentation-editor.ts:62`, `tiptap-mini-editor.ts:48`
**PR:** #491

`handleContentUpdate` writes `editor.getHTML()` straight into Yjs and re-renders by feeding the string back to `new Editor({ content })`. TipTap's StarterKit + Underline schema does **not** configure URL/attribute sanitization for `href` / event handlers. Any future renderer that does `el.innerHTML = content` (thumbnail generator, exporter, federated KB, RAG context, email summary) becomes stored XSS. The contract makes no statement that content is sanitized.

**Fix:** add `sanitizeRichTextHtml(html)` (DOMPurify or a strict allowlist over the StarterKit+Underline node/mark set) and call it on **both** write *and* read paths. Add invariant to `contracts/app-slides/rules.md`.

### C3 — Slides: tables have no row/col bounds (memory DoS)
**File:** `modules/app-slides/contract.ts:21`, `internal/render-table.ts:20-37`, `element-factory.ts:110`
**PR:** #490

`rows: z.number().int().positive()` accepts `2_000_000_000`. `createTableElement(rows, cols)` is exported. A collaborator can write `rows: 1e9, cols: 1e9` directly via `Y.Map.set('tableData', JSON.stringify({rows: 1e9, cols: 1e9, cells: []}))`. Every peer's `parse-elements.ts:39` then attempts to render. Browser is bricked across the whole collab session.

The insert toolbar UI caps at 6×6, but that's UX-only; the data layer is unprotected.

**Fix:** `MAX_TABLE_ROWS = 50`, `MAX_TABLE_COLS = 20` in `contract.ts`. Apply via `.max(50)`/`.max(20)` to the schema, clamp/reject in `createTableElement`, `parseTableData`, `parse-elements.ts`. Cap each cell string at 8 KB.

### C4 — SSE leaks Grant events to every authenticated user (privacy/IDOR oracle)
**File:** `modules/api/internal/sse-routes.ts:63-74`
**PR:** #501

The handler comment explicitly says the GrantCreated/GrantRevoked stream is broadcast to all SSE subscribers and "the client filters." Any authenticated user can connect and observe in real time:

- every grantId being created/revoked,
- the docIds involved,
- actor identities,
- frequency / timing patterns.

This is a global activity oracle. Server-side filtering must use `event.granteeId` and `event.actorId` to gate forwarding to the connected principal.

**Fix:** in the SSE handler, only forward Grant events when `granteeId === principalId` OR `actorId === principalId`. Same pattern for any future event type carrying access metadata.

### C5 — SSE `allowedDocIds` is frozen at connect; revocations don't tighten
**File:** `modules/api/internal/sse-routes.ts:42-48`
**PR:** #501

The Set of allowed docIds is built once at connection time. If a grant is revoked while a user holds the stream, they keep receiving DocumentUpdated/StateFlushed for that doc until they disconnect. Mirrors the WS issue that `grant-revoked-handler.ts` fixed for the WS path — but SSE wasn't included.

**Fix:** subscribe to GrantRevoked-for-this-principal inside the SSE handler. On revoke, `allowedDocIds.delete(docId)` and force-close the response. On GrantCreated for self, `add()`.

### C6 — Pending-grant activation trusts unverified email (invite hijack)
**File:** `modules/sharing/internal/activate-pending-grants.ts:20-66`, `migrations/016_pending_grants.sql`
**PR:** #501

Activation runs against `(userId, email, ...)` with no check that the email was verified by the IdP. Any auth provider that accepts unverified email claims (and most do by default unless explicitly forced to require `email_verified=true`) lets an attacker register `victim@example.com` and inherit every pending invite ever sent to that address. There is also no unique index on `(doc_id, grantee_email)` filtered to `status='pending'`, so duplicate invites stack.

**Fix:** require `principal.emailVerified === true` before calling `activatePendingGrants`. Lowercase + canonicalize email on both sides. Add unique index. Long-term: switch to signed-link redemption (grantId in token, no email matching needed).

### C7 — TOCTOU on pending-grant role ceiling
**File:** `modules/sharing/internal/routes-invite.ts:60-73`, `activate-pending-grants.ts:38-44`
**PR:** #501

The role-ceiling check happens at **invite time** (reads grantor's current role) but the grant is created at **acceptance time** (uses the cached role from the pending row). If the grantor is later demoted or fully revoked before the invitee accepts, the system still grants the invitee the originally-promised (now-illegitimate) role.

**Fix:** at activation, re-resolve grantor's current role on `docId` and clamp `grant.role` to `min(grant.role, grantorCurrent)`. If the grantor has been fully revoked, drop the pending grant.

### C8 — Cold-tier read path bypasses RLS (IDOR)
**File:** `modules/storage/internal/pg-document-repository.ts:84-118`, `cold-storage.ts:48-119`
**PR:** #501

`getSnapshot` and the cold-warm-up path call `pool.query` directly instead of `rlsQuery`. Per the comment in `pg-pending-grant-store.ts:5-12`, RLS is the access boundary for this codebase. Therefore the cold tier returns any document by id regardless of caller principal — full IDOR if any HTTP path reaches this without an upstream permission check.

**Fix:** route all reads through `rlsQuery`, OR assert that callers always pre-check via `permissions.checkPermission`. Add a contract test that asserts a non-grant-holder cannot read a cold-tier doc.

### C9 — Pivot DoS via `Math.min/max(...values)` spread
**File:** `modules/app-sheets/internal/pivot/pivot-aggregations.ts:37-38`
**PR:** #497

`Math.min(...values)` and `Math.max(...values)` spread arrays into argument lists. With ~125k+ values, V8 throws `RangeError: Maximum call stack size exceeded`. A 1M-row dataset is realistic for a pivot use case — this crashes deterministically.

**Fix:** replace with a `for`-loop reduce.

### C10 — Pivot ignores user-selected range (data leak)
**File:** `modules/app-sheets/internal/pivot/pivot-engine.ts:139`
**PR:** #497

`parseSourceRange(_rangeStr, sheetData)` discards the `rangeStr` parameter and aggregates ALL sheet data. If a sheet contains both pivot-source rows and an unrelated protected column, the pivot silently aggregates across the boundary — leaking values the user didn't intend to expose. Also balloons memory when sheets are large.

**Fix:** parse and apply `rangeStr`. Reject pivot creation if the range can't be parsed.

### C11 — #485 fix landed in the wrong file; original vulnerability still exploitable
**File:** `modules/kb/internal/kb-markdown-helpers.ts:60-74` (still vulnerable)
**PR:** #495 (incomplete)

The PR patched `modules/convert/internal/html-parser*` (Collabora/LibreOffice import path), not the originally-flagged `parseConfluenceHtml` in `kb-markdown-helpers.ts`. Lines 70-71 still execute `stripTags` BEFORE `decodeEntities`:

```js
.replace(/<[^>]+>/g, '')                        // line 70: strip
.replace(/&amp;/g, '&').replace(/&lt;/g, '<')…  // line 71: decode
```

PoC: upload an HTML file containing `<p>&lt;script&gt;alert(1)&lt;/script&gt;</p>`. Tag-strip is a no-op; decode produces literal `<script>alert(1)</script>` stored in `metadata.body`. The pre-existing 2026-04-15 review flagged this exact line and it was not touched. The added test in `html-parser-marks.test.ts:107-153` covers a parallel path, giving false confidence.

Today, the SPA renderer (`simple-markdown.ts` `escapeHtml`, `entry-card.ts` `textContent`) neutralizes on display — but the bug as titled ("storing executable markup") is unfixed. Any future export consumer, federated KB share, or HTML email preview triggers it.

**Fix:** in `kb-markdown-helpers.ts:60-74`, decode entities BEFORE stripping tags, OR (better) use the same parser the convert/ path now uses.

## High Findings

### H1 — #484 fix is incomplete: KB public listing exposes draft content
**File:** `modules/kb/internal/kb-public-routes.ts:65-87`, `kb-routes.ts:102`
**PR:** #495

The toggle (`POST /api/kb/settings`) now requires admin. But the public listing returns ALL entries regardless of lifecycle status — `status='draft'`, `status='unreviewed'`, anything. Once an admin flips the bit, unauthenticated visitors see in-progress and unreviewed content with whatever PII or stored-XSS payloads exist. Additionally, `POST /api/kb/:id/transition` is gated only by `requireAuth`, so any logged-in non-admin can move arbitrary entries to `published`.

**Fix:** filter the public listing on `status='published'` (use `listPublishedEntries` or equivalent). Restrict `POST /api/kb/:id/transition` to entry owner OR admin.

### H2 — `pruneStateVector` corrupts Yjs state vectors (data loss after 30 days)
**File:** `modules/storage/internal/pg-document-repository.ts:26-42`
**PR:** #501

```js
Y.encodeStateVector(decoded as unknown as Y.Doc)
```

`encodeStateVector` expects a `Y.Doc`, not a `Map<number, number>`. The unsafe cast produces garbage bytes. After 30 days the doc's state vector becomes unreadable, breaking diff sync.

**Fix:** reconstruct via `encoding.writeVarUint`-based encoding, or simply `Buffer.from(raw)` until a correct prune is implemented.

### H3 — `archiveToCold` and the matching SQL UPDATE are non-atomic
**File:** `modules/storage/internal/cold-storage.ts:140-142`
**PR:** #501

`s3.send(PutObjectCommand)` runs first, then `pool.query(UPDATE … snapshot=NULL)`. If PUT succeeds but UPDATE crashes, you have a phantom cold object the doc doesn't know about; if UPDATE succeeds but a later read fails warm-up, the snapshot is already nulled — data loss. `archiveStaleDocuments` swallows errors with `console.error`, so this fails silently.

**Fix:** S3 first → verify GET works → then UPDATE in a transaction with `tier='cold'` only. Use a journal row to track in-flight archives.

### H4 — `warmFromCold` deserializes attacker-controlled JSON unchecked
**File:** `modules/storage/internal/cold-storage.ts:88-118`
**PR:** #501

`JSON.parse(raw)` is cast directly to `ColdObject` and re-inserted into the documents table. If S3 contents are ever tampered with (compromised bucket, wrong region, key collision), this becomes blind JSON injection into hot storage with no schema check.

**Fix:** define a Zod schema for `ColdObject` and reject on mismatch.

### H5 — Invite endpoint has no rate limiting (spam relay + email enumeration)
**File:** `modules/sharing/internal/routes-invite.ts:42-112`
**PR:** #501

`addInviteRoute` has zero rate limit, unlike `routes.ts` which has `resolveRateLimiter`. An attacker with `write` on any doc can spray invites at arbitrary email addresses. The InviteCreated event triggers email by design (line 82-99), so this turns OpenDesk into a spam relay. Pending-grant rows accumulate unbounded.

**Fix:** per-grantorId and per-docId rate limits. Unique index on `(doc_id, grantee_email)` filtered to `status='pending'`.

### H6 — Role-ceiling check has a fall-through to rank 0
**File:** `modules/sharing/internal/routes-invite.ts:65-69`
**PR:** #501

`bestRank < (ROLE_RANK[role] ?? 0)`. If a typo or future role addition leaves an entry missing from `ROLE_RANK`, the `?? 0` fallback makes the comparison trivially true and **any grantor passes the ceiling check**. Ranks should never silently default.

**Fix:** assert `ROLE_RANK[role] !== undefined`. Throw 500 if not.

### H7 — Slides `parseElement` silently corrupts bad data instead of rejecting
**File:** `modules/app-slides/internal/parse-elements.ts:12-43`
**PR:** #490

Unknown `type` values silently coerce to `text`-shaped output but pass through with the original `type` field intact. The renderer's default branch then re-recurses with `{ ...el, type: 'text' }`. A non-string `id` or non-numeric `x` becomes `String(undefined)`. The exported `SlideElementSchema` is decorative — `parseElement` never calls `.parse()` / `.safeParse()`.

**Fix:** validate every element with `SlideElementSchema.safeParse` and drop non-conforming maps. Log + skip.

### H8 — Slides: Yjs `applyFieldUpdate` writes arbitrary fields with no allowlist
**File:** `modules/app-slides/internal/yjs-mutations.ts:89`
**PR:** #490

`applyFieldUpdate(yel, field: string, value)` writes whatever `field` is given. A malicious peer can call `yel.set('__proto__', ...)` or write 100 MB blobs into unread fields that all peers must replicate. `parse-elements.ts` only reads named fields, so unknown keys persist on the `Y.Map` forever, surviving collab round-trips.

**Fix:** define `ALLOWED_ELEMENT_FIELDS` per element type. Reject unknown fields. Periodic prune of unknown keys on read.

### H9 — Slides: `updateTableCell` writes unbounded `value` and bypasses rows/cols bounds
**File:** `modules/app-slides/internal/yjs-element-insert.ts:55-73`
**PR:** #490

The handler checks `tableData.cells[row]` exists, but JavaScript happily extends a sparse array on out-of-range writes. A peer can call `updateTableCell(..., 9_999_999, 0, 'x')` to bloat the JSON serialization. No length cap on `value`.

**Fix:** bounds-check `row < tableData.rows && col < tableData.cols` before writing. Cap `value.length`.

### H10 — Formula range expansion has no size cap (memory bomb)
**File:** `modules/sheets-formula/internal/evaluator.ts:44-59` (`expandRange`), `circular-detect.ts:26-27`
**PR:** #496

`=SUM(A1:ZZZ1048576)` materializes ~50M cell-reference strings. Tab freeze.

**Fix:** cap `(maxCol-minCol+1) * (maxRow-minRow+1)` at 1M. Return `#REF!` past that.

### H11 — Recursive parser + recursive evaluator have no depth limit (stack overflow)
**File:** `modules/sheets-formula/internal/parser.ts`, `evaluator.ts:68`, `circular-detect.ts:64`
**PR:** #496

`=((((((((...)))))))) ` ~10k deep blows the JS stack. `dfs()` recursion across long dependency chains is also unbounded.

**Fix:** add a depth counter, abort with `#VALUE!` / `#REF!` past 256.

### H12 — ReDoS in SEARCH and XLOOKUP wildcard matching
**File:** `modules/sheets-formula/internal/functions-text-ext.ts:66-72`, `evaluator-xlookup.ts:139-141`
**PR:** #496

User-controlled wildcard `*…*…*` becomes `.*.*.*` regex. Catastrophic backtracking against long `withinText`. XLOOKUP compiles the regex inside the per-row loop.

**Fix:** convert to a non-regex two-pointer wildcard match, OR anchor patterns + decompose with `String.prototype.includes`. Compile once outside the loop.

### H13 — REPT memory bomb
**File:** `modules/sheets-formula/internal/functions-text-ext.ts:93`
**PR:** #496

`text.repeat(Math.floor(times))` with `=REPT("ab", 1e9)` throws or eats GBs.

**Fix:** cap result length at 32,767 (Excel parity). Return `#VALUE!` past that.

## Medium Findings

### M1 — IntentExecutor has no per-doc serialization; idempotency cache is unscoped; no permission check
**File:** `modules/collab/internal/intent-executor.ts:88,106-153`
**PR:** #501

Three connected issues:
1. Two concurrent `applyIntent` calls on the same docId both pass the baseRevision check and both transact. The second writer's revisionId clobbers the first's idempotency cache entry, breaking `DUPLICATE_INTENT` semantics.
2. Idempotency cache key is `idempotencyKey` alone — a string the *client* chose. Two users with overlapping keys (or attacker guessing) get cross-talk: replay returns *another user's* `revisionId`. Scope as `${principalId}:${documentId}:${idempotencyKey}`.
3. `applyIntent` accepts a `DocumentIntent` and mutates the Y.Doc with no permission check. Whatever HTTP route is later added MUST enforce write — but the current export is a footgun. The action payloads in `intent-executor-spreadsheet.ts:4` cast `Record<string, unknown>` and skip Zod validation; attacker-supplied `action.row` / `action.col` can be huge ints causing OOM in `Y.Array.delete(2e9, 1)`.

**Fix:** per-docId mutex (e.g. `p-queue` keyed by docId). Compound idempotency key. Add `principal: Principal` parameter to `applyIntent` plus an entry-side permission check. Zod-validate action payloads.

### M2 — Data validation is purely advisory; "custom" type is a no-op stub
**File:** `modules/app-sheets/internal/data-validation/engine.ts:40-41`, `renderer.ts:48`
**PR:** #494

`validate()` for `type='custom'` returns `{valid: true}` unconditionally. `applyValidationIndicators` only adds CSS classes — it never blocks a write, never rolls back, never rejects. Paste, autofill, formula results, and direct Yjs edits all bypass. No server-side enforcement anywhere. The `errorStyle: 'reject'` UI hint is a lie.

**Fix:** add a write-side gate in `onCellValueChange` (and any other mutator) that calls `validate()` and rejects when `errorStyle === 'reject'`. Implement custom-formula evaluation via the formula evaluator. Apply server-side via the same intent-validation layer when intents are wired.

### M3 — BibTeX parser ReDoS + unbounded body translates to unbounded memory
**File:** `modules/references/internal/bibtex-parser.ts:33-40`, `reference-import-routes.ts:60-96`
**PR:** #501

`extractFieldValue` is a hand brace walker with O(n²) worst case on deeply nested braces. Body parsing happens before the 500-entry cap, so a 1 MB body of pathological `{{{{...}}}}` burns CPU before the cap triggers.

`parseFields` does `fields[m[1].toLowerCase()] = …` on a plain object literal. A field named `__proto__` sets `Object.prototype.__proto__`. Use `Object.create(null)`.

**Fix:** cap raw body bytes at 256 KB on the route. Add a parse-time entry budget. Use `Object.create(null)` for fields.

### M4 — `archiveStaleDocuments` LIMIT 100 with no offset → starvation
**File:** `modules/storage/internal/cold-storage.ts:128-143`
**PR:** #501

If 100 docs always fail to archive, no other docs ever get archived.

**Fix:** randomize order or track skip-on-error.

### M5 — Materializer flush emits two `eventBus.emit` calls non-atomically
**File:** `modules/collab/internal/document-materializer.ts:140-151`
**PR:** #501

DocumentUpdated can succeed while StateFlushed fails.

**Fix:** bundle into a single outbox transaction.

### M6 — Slides `insert-toolbar` document-level click listeners leak on remount
**File:** `modules/app-slides/internal/insert-toolbar.ts:95,125`
**PR:** #490

`document.addEventListener('click', ...)` with no `removeEventListener` on teardown. Each editor remount leaks two document-level listeners holding closures over the toolbar wrapper.

**Fix:** return `destroy()` from `createInsertToolbar`. Editor's `destroy()` invokes it.

### M7 — Slides `parse-elements` defaults are silently corrective
**File:** `modules/app-slides/internal/parse-elements.ts:16-21`
**PR:** #490

`(el.get('width') as number) || 50` triggers on `width: 0` and replaces with `50`. Conflict-resolution bugs upstream become invisible. Use `?? 0` for nullish-only defaults.

### M8 — Slides `parseTableData` doesn't validate `cells` shape
**File:** `modules/app-slides/internal/yjs-element-insert.ts:79-89`
**PR:** #490

Only checks `Array.isArray(parsed.cells)`. Inner items can be anything. Later `String(...)` of `[{toString(){fetch(...);return ''}}]` runs network code at render time.

**Fix:** Zod-validate `cells: z.array(z.array(z.string().max(8000)))`. Ensure inner row length ≤ `cols`.

### M9 — Slides image upload spawns parallel uploads with no concurrency cap
**File:** `modules/app-slides/internal/slide-image-upload.ts:53-55`
**PR:** #490

Drop 50 files → 50 simultaneous POSTs. Easy DoS on the upload endpoint.

**Fix:** serialize uploads or use a small p-limit pool.

### M10 — Pivot total computation is O(R·C·V) with no memory cap
**File:** `modules/app-sheets/internal/pivot/pivot-engine.ts:95-115, 117-137`
**PR:** #497

A 1M-row × 1k-rowKeys × 100-colKeys pivot is multi-GB.

**Fix:** aggregate incrementally during the first pass instead of recomputing from `rawCells`.

### M11 — Financial functions divide-by-zero leaks Infinity
**File:** `modules/sheets-formula/internal/functions-financial.ts:25-26 (PMT), 34-35 (FV), 43-44 (PV)`
**PR:** #496

With `rate = -2`, `pvif = 1`, denominator `(pvif-1)/rate === 0`. PMT returns `±Infinity` instead of `#NUM!`. Same for IRR `Math.pow(1+rate, i)` with `rate = -1`.

**Fix:** detect non-finite results, return `#NUM!`.

### M12 — `svg-builder` `esc()` is incomplete; raw interpolation in path/extra/transform
**File:** `modules/sheets-chart/internal/svg-builder.ts:3-9, 21, 45, 56`
**PR:** #498

`esc()` does not escape `>` or `'`. The `extra` parameter on `rect()`, the `d` parameter on `path()`, and the `rotate` value in `text()` are interpolated raw. Currently safe in practice (literal/numeric callers only), but a single refactor that reaches user content here is stored XSS.

**Fix:** add `>` and `'` to `esc()`. Wrap `extra` through a sanitizer or remove the escape hatch. Type `rotate: number` and explicitly `r()` it.

### M13 — Slides contract docs lack invariants for new element types
**File:** `contracts/app-slides/rules.md`
**PR:** #490, #491

No invariant for image src allow-listing, rich-text sanitization, or table bounds. Future agents reading the contract won't know these are required.

**Fix:** add invariants 11-13: "Image src must be http(s) or relative `/uploads/`", "Rich text content is sanitized on write and read", "Tables clamped to MAX_TABLE_ROWS × MAX_TABLE_COLS".

If the collab WS server and the grant-revoke emitter run in different pods, there's a window between `grantStore.update(status='revoked')` and `subscribeGrantRevoked` closing the socket where the user keeps editing. Pair with a per-message permission check on incoming Yjs updates as defense in depth — `server.ts` only authenticates at connect.

5. **SlideElementSchema is decorative.** Defined and exported, but never enforced. Any time a Zod schema is exported but no caller `.safeParse()`s it, that's an accident waiting to happen.

6. **#485 fix landed in a different file than the originally-flagged one.** Process gap — the 2026-04-15 review referenced `kb-markdown-helpers.ts:60-74` by line, and the fix shipped in `modules/convert/`. Suggest tightening: every security fix PR should quote the file:line from the original report and link to it.

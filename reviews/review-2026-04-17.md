## High Findings

### H1 — `runStep`: `with-previous` / `after-previous` semantics inverted
**File:** `modules/app-slides/internal/animation-engine.ts`, `runStep` (~line 100)
**Issue:** #486

The lane-building logic is the mirror image of the spec. `after-previous` opens a **new parallel lane** (runs simultaneously with other lanes via `Promise.all`). `with-previous` stays in the **same sequential lane** (runs only after the previous item finishes). The decision log and contract both state the opposite: `with-previous` is the parallel case, `after-previous` is the sequential case.

```ts
// Current (wrong):
if (i === 0 || a.trigger === 'after-previous') lanes.push([a]);  // new = parallel
else lanes[lanes.length - 1].push(a);                            // same = sequential

// Correct:
if (i === 0 || a.trigger === 'with-previous') lanes.push([a]);   // new = parallel
else lanes[lanes.length - 1].push(a);                            // same = sequential
```

The existing test (`after-previous animations run sequentially within a step`) passes coincidentally because it only checks invocation order, not causal dependency. A corrected test must assert that animation #2's `animate()` is not called until animation #1's `onfinish` fires.

**Impact:** Any presenter deck using `with-previous` or `after-previous` plays animations in the wrong order / wrong concurrency model.

### H2 — `initAnimations()` return value discarded — resource leak
**File:** `modules/app-slides/internal/presentation-editor.ts` (~line 148, 164)
**Issue:** #487

`initAnimations()` returns an `AnimationInitResult` containing `destroy()`. The caller drops the return value, so the Yjs `observeDeep` listener and two DOM nodes (the `<aside>` panel and the toolbar `<button>`) are never cleaned up when the editor is destroyed.

```ts
// presentation-editor.ts — fix
const animResult = initAnimations({ ... });

return {
  destroy() {
    animResult.destroy();   // ← missing
    interactionCtrl?.destroy();
    provider.destroy();
    ydoc.destroy();
  },
};
```

**Impact:** Each editor mount/unmount cycle leaks an observer and DOM nodes. In a SPA with navigation, this accumulates unboundedly.

## Medium Findings

### M1 — `reset()` does not cancel in-flight Web Animations API handles
**File:** `modules/app-slides/internal/animation-engine.ts`, `runAnimation` and `reset`

`runAnimation` uses `fill: 'forwards'` on the Web Animations API call and never stores the returned `Animation` handle. `reset()` calls `prepareInitialState` to re-hide entrance elements, but any animations still running from a previous `next()` call will fire their `onfinish` callback after the reset, overwriting the freshly-hidden state (setting `opacity: '1'` on an element that was just re-hidden to `'0'`).

Reproduce: in presenter mode, navigate back to a slide while its entrance animations are mid-play, then advance through them again — some elements flash visible before their entrance animation runs.

**Remediation:** Store handles returned by `el.animate()` and call `.cancel()` on each during `reset()`.

## Low Findings

### L1 — `appendAnimation` has no runtime validation of `effect`
**File:** `modules/app-slides/internal/animation-yjs.ts`, `appendAnimation`

`updateAnimationField` guards against invalid effect names with `isAnimationEffect(value)` but `appendAnimation` trusts the caller's TypeScript type. Non-TypeScript callers or future JS interop could store an invalid effect that `parseAnimation` would then silently skip, making the animation disappear from the list with no error.

**Remediation:** Add `if (!isAnimationEffect(init.effect)) throw new Error(...)` (or a guard + early return) to mirror `updateAnimationField`.

---

### L2 — `moveAnimation` preserves arbitrary extra Y.Map fields
**File:** `modules/app-slides/internal/animation-yjs.ts`, `moveAnimation`

The snapshot-and-rebuild approach copies all keys from each Y.Map row without filtering to the known schema. In a collaborative session, a malicious or buggy peer could inject extra fields that would survive reorder operations indefinitely. While there is no code-execution risk (the extra fields are ignored by `parseAnimation`), they bloat the Yjs document and may interfere with future schema evolution.

**Remediation:** Whitelist fields in the rebuild loop: `['id', 'elementId', 'effect', 'trigger', 'durationMs', 'delayMs']`.

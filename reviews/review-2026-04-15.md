## Critical Findings

### CRIT-1: Model zoo management routes have no authentication
**File:** `modules/ai/internal/ai-routes.ts` lines 74–127  
**GitHub issue:** 1889ca/opendesk#483

All seven routes in the `if (modelService)` block are registered without any auth middleware:

```
GET  /api/ai/models
POST /api/ai/models/:id/pull
DEL  /api/ai/models/:id
GET  /api/ai/models/:id/status
PUT  /api/ai/config
POST /api/ai/models/custom
DEL  /api/ai/models/custom/:id
```

Any unauthenticated HTTP request can:
- Trigger arbitrary Ollama model downloads (disk exhaustion, SSRF to Ollama)
- Delete installed models (DoS against AI features)
- Overwrite the active embedding/generation model for all workspace users
- Register a custom model pointing to an attacker-controlled endpoint (SSRF)

The RAG routes immediately above (lines 30–70) correctly use `permissions.requireAuth`. The model service block was written separately and the auth middleware was omitted entirely.

**Remediation:** Add `permissions.requireAuth` (and an admin-scope check) to every route in the `modelService` block.

## High Findings

### HIGH-1: Any authenticated user can toggle KB public access (missing role check)
**File:** `modules/kb/internal/kb-public-routes.ts` lines 103–121  
**GitHub issue:** 1889ca/opendesk#484

`POST /api/kb/settings` only checks `requireAuth`. A viewer invited via share link can set `is_public: true`, exposing the entire knowledge base unauthenticated; or set `is_public: false` to disrupt a public workspace. This should be restricted to admin/owner principals.

**Remediation:** Add an admin scope or owner-role guard after `requireAuth` on the settings mutation endpoint.

---

### HIGH-2: Confluence HTML import decodes entities after tag-stripping, storing executable markup
**File:** `modules/kb/internal/kb-markdown-helpers.ts` lines 60–74  
**GitHub issue:** 1889ca/opendesk#485

`parseConfluenceHtml` strips HTML tags with `/<[^>]+>/g` and **then** decodes HTML entities (`&lt;` → `<`). Entity-encoded tags (e.g. `&lt;script&gt;`) are invisible to the regex stripper and survive to be decoded as literal `<script>` before database storage. If any KB rendering path passes the stored body through a markdown renderer with HTML passthrough enabled, this is stored XSS.

**Remediation:** Decode HTML entities **before** stripping tags so the stripper catches decoded markup. Consider replacing regex parsing with a proper sanitizer (`sanitize-html`, `DOMPurify` via `linkedom`).

## Medium Findings

### MED-1: No limit on ZIP entry count during KB import (DoS)
**File:** `modules/kb/internal/kb-import-export-routes.ts` lines 90–114, `modules/kb/internal/kb-zip.ts` lines 127–156

The 50 MB multer limit bounds raw file size, but a crafted ZIP with many small `.md` files (e.g. 50,000 × 1 KB entries) can trigger 50,000 sequential `createEntry` database inserts per request. This will exhaust DB connection pool time and could time out or degrade service for other users.

**Remediation:** Add a per-import entry count limit (e.g. 500 files) in `extractMdFiles` or early in the import handler. Return HTTP 400 if exceeded.

---

### MED-2: Public KB endpoint returns `corpus` field without filtering by corpus type
**File:** `modules/kb/internal/kb-public-routes.ts` lines 64–87

The public read endpoint returns all entries including those with `corpus: 'operational'` or `corpus: 'reference'`. Operational entries may contain internal process knowledge not intended for public exposure. The filter should restrict public access to `corpus: 'knowledge'` entries only, or require explicit per-entry opt-in for public visibility.

**Remediation:** Filter `listEntries` results to `corpus === 'knowledge'` (or a dedicated `is_public_entry` flag) before returning from the public route.

## Low Findings

### LOW-1: ZIP import filenames not sanitized before use as entry titles
**File:** `modules/kb/internal/kb-zip.ts` lines 140–146, `modules/kb/internal/kb-import-export-routes.ts` lines 92–95

File names extracted from ZIP headers (e.g. `../../etc/shadow.md`) are used as the basis for entry titles without sanitizing path separators. While no filesystem writes occur, the resulting titles can contain confusing or adversarial content. `sanitizeFilename` is correctly applied at export time but not on import.

**Remediation:** Apply `sanitizeFilename` to the `name` field returned by `parseZip` before using it as a title source.

---

### LOW-2: MIME type spoofing can bypass file-type routing in KB import
**File:** `modules/kb/internal/kb-import-export-routes.ts` lines 28–39

File type detection in `extractMdFiles` uses both `originalname` suffix and user-supplied `mimetype` (from the Content-Type header). An attacker can set `Content-Type: text/plain` on a ZIP payload with a `.md` extension to bypass ZIP parsing and have the raw ZIP binary treated as markdown text. The practical impact is low (corrupt import, not a security escalation), but the detection logic should prefer the filename extension and ignore the MIME type for routing decisions.

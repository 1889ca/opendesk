## High Findings

### HIGH-1 — IDOR: Starred-documents endpoint accepts any document ID without a read-permission check

**File:** `modules/document/internal/starred-routes.ts`, lines 36–51  
**Migrated from:** `migrations/007_starred_documents.sql` (FK `document_id REFERENCES documents(id)`)

**Description**

`POST /api/starred/:documentId` only requires `permissions.requireAuth`.  It does **not** call `permissions.require('read')` or verify that the authenticated principal holds any grant on the target document.

Because `starred_documents.document_id` is a foreign key to `documents.id`, the database will:

- Return **201** when the UUID refers to a document that exists.
- Throw a FK violation (→ unhandled 500) when the UUID does not match any document.

This gives any authenticated principal a reliable oracle to determine whether **any** document UUID exists — regardless of whether they have read, viewer, or any other grant on that document.  Combined with the fact that the principal's starred list subsequently queries their own rows, it is possible to build up a set of valid document IDs without ever being granted access.

**Proof of concept**

```
POST /api/starred/<any-valid-uuid>
Authorization: Bearer dev:<attacker-id>:Attacker:*

→ 201  (document exists, attacker now knows)
→ 500  (document does not exist)
```

**Remediation**

Add `permissions.require('read')` as a middleware before the star handler — the same pattern already used by `GET /:id` in `document-mutation-routes.ts`:

```typescript
router.post(
  '/:documentId',
- permissions.requireAuth,
+ permissions.require('read'),   // ← enforce read grant on the document
  asyncHandler(async (req, res) => { ... }),
);
```

The `DELETE /:documentId` handler is safe (it only deletes the caller's own row and returns 404 when no star exists — no information leak about documents the caller did not star themselves).

## Medium Findings

### MED-1 — Reflected client-supplied HTML served as `text/html` in export route

**File:** `modules/document/internal/export-routes.ts`, lines 33–48

**Description**

`POST /:id/export` with `format=html` reads `req.body.content` — a **client-supplied string** — and sends it back with `Content-Type: text/html; charset=utf-8`.  The content is not fetched from the database; it is whatever the caller included in the POST body.

```typescript
const output = format === 'html' ? content : /* strip tags */;
res.setHeader('Content-Type', 'text/html; charset=utf-8');
res.setHeader('Content-Disposition', `attachment; filename="..."`);
res.send(output);   // ← arbitrary HTML echoed from request body
```

`Content-Disposition: attachment` is the only defence against browser rendering.  Most modern browsers honour this, but:

1. Older or misconfigured browsers (IE, some WebView embeddings) may ignore it.
2. If a CSRF attack is crafted to call this endpoint on behalf of a victim, the downloaded file could be named and distributed as a malicious HTML payload.
3. The endpoint only requires `read` access, meaning any viewer or share-link redeemer can trigger it.

**Remediation**

Export should be derived from the server-side document state (Yjs / `content_plain`), not echoed from the request body.  If client-driven export is required, validate and sanitise the HTML with a server-side allowlist before sending (e.g. DOMPurify on the server, or a Cheerio-based sanitiser).  At minimum, add `X-Content-Type-Options: nosniff` on this specific response (helmet sets it globally, so verify it is not being overridden).

### MED-2 — Path traversal: `startsWith` without trailing-slash in HTML serving middleware

**File:** `modules/api/internal/csp-nonce.ts`, line 54

**Description**

```typescript
const filePath = resolve(publicDir, reqPath.replace(/^\//, ''));
if (!filePath.startsWith(publicDir)) return next();
```

`publicDir` is resolved without a trailing slash (e.g. `/app/modules/app/internal/public`).  A crafted path that escapes the directory but resolves to a string that **shares the prefix** — e.g. `/app/modules/app/internal/public-other/secret.html` — would pass the `startsWith` guard and attempt to read a file outside the intended directory.

In the current Docker deployment nginx normalises `..` sequences before they reach Express, making exploitation unlikely in practice.  Nevertheless the check violates defence-in-depth.

**Remediation**

```typescript
- if (!filePath.startsWith(publicDir)) return next();
+ if (!filePath.startsWith(publicDir + '/') && filePath !== publicDir) return next();
```

### MED-3 — Document list fetches up to 10 000 rows and filters grants in application memory

**File:** `modules/document/internal/document-routes.ts`, lines 94–105

**Description**

```typescript
const allResult = await listDocuments({
  ...
  limit: 10000,  // ← fetches entire visible corpus
  offset: 0,
});
const filtered = allResult.rows.filter((doc) => allowedIds.has(doc.id));
```

The route fetches the entire document table (capped at 10 000 rows) on every list request, then discards rows the principal does not have access to.  On a moderately busy instance this means:

1. **Performance**: a single list request may transfer and deserialise tens of megabytes of rows.
2. **DoS amplification**: an authenticated user with minimal grants can trigger large sequential scans on every page load.
3. **Correctness cliff**: once the corpus exceeds 10 000 documents the result is silently truncated — granted documents beyond the cap become invisible.

**Remediation**

Push the grant filter into the SQL query using a JOIN or `WHERE id = ANY(...)` — the same pattern already used by `searchDocuments` and `globalSearch`.  Pagination can then happen at the database level.

```sql
SELECT d.* FROM documents d
JOIN grants g ON g.resource_id = d.id
  AND g.resource_type = 'document'
  AND g.principal_id = $1
WHERE ...
ORDER BY ...
LIMIT $N OFFSET $M
```

### MED-4 — Global search skips permission filtering in dev mode; inconsistent with `search-routes.ts`

**File:** `modules/document/internal/global-search-routes.ts`, lines 52–58

**Description**

```typescript
if (loadConfig().auth.mode !== 'dev') {
  const grants = ...;
  allowedIds = grants.filter(...).map(...);
}
let results = await search(parsed.data.q, allowedIds);  // undefined → no filter
```

When `AUTH_MODE=dev`, `allowedIds` remains `undefined` and `globalSearch` returns **all documents**.  The regular per-document search route (`search-routes.ts`) removed this bypass explicitly in issue #66 with the comment *"Skipping this was an IDOR vulnerability"*.  The global search route was not updated at the same time.

Additionally, `loadConfig()` is called on every request, re-reading environment variables on each search.

**Remediation**

Remove the mode guard; always resolve `allowedIds`, mirroring `search-routes.ts`:

```typescript
- if (loadConfig().auth.mode !== 'dev') {
-   const grants = ...;
-   allowedIds = ...;
- }
+ const grants = await permissions.grantStore.findByPrincipal(principal.id);
+ const allowedIds = grants
+   .filter((g) => g.resourceType === 'document')
+   .map((g) => g.resourceId);
```

## Low Findings

### LOW-1 — `Content-Disposition` filename uses `encodeURIComponent` inside a quoted string

**Files:**
- `modules/document/internal/export-routes.ts`, line 44
- `modules/convert/internal/convert-routes.ts`, line 125
- `modules/convert/internal/presentation-convert-routes.ts`, line 131

**Description**

```typescript
res.setHeader('Content-Disposition', `attachment; filename="${encodeURIComponent(fname)}"`);
```

`encodeURIComponent` produces percent-encoded strings (e.g. spaces → `%20`).  Inside a `filename=` quoted-string parameter, percent-encoding is not decoded by all HTTP clients.  The RFC 6266 / RFC 5987 standard approach for non-ASCII filenames is `filename*=UTF-8''<rfc5987-encoded>`.  Browsers that strictly follow the spec may display the literal `%20` in the saved filename.

**Remediation**

Use the `filename*` form for non-ASCII names and strip/replace characters that are invalid in `filename=` quoted strings:

```typescript
const safeAscii = fname.replace(/[^\x20-\x7E]/g, '_').replace(/[\\"/]/g, '_');
res.setHeader(
  'Content-Disposition',
  `attachment; filename="${safeAscii}"; filename*=UTF-8''${encodeURIComponent(fname)}`,
);
```

### LOW-2 — Fire-and-forget `updateContentPlain` in preview route may silently lose updates

**File:** `modules/document/internal/preview-routes.ts`, line 47

**Description**

```typescript
if (preview) {
  updateContentPlain(documentId, preview).catch(() => {});
}
```

Database errors (transient connectivity, pool exhaustion) are swallowed.  If the update fails, `content_plain` remains stale.  Subsequent full-text searches will miss or misrank the document.  There is no log entry and no retry, making silent failures invisible in observability tooling.

**Remediation**

Log the error at `warn` level so it surfaces in monitoring:

```typescript
updateContentPlain(documentId, preview).catch((err) => {
  log.warn('updateContentPlain failed', { documentId, error: String(err) });
});
```

## Critical Findings

### CRIT-1: Collab WebSocket — No Document-Level Permission Check (IDOR)

**File:** `modules/collab/internal/authenticate.ts`  
**Severity:** CRITICAL

The Hocuspocus `onAuthenticate` hook verifies the user's bearer token but **never checks whether the authenticated user has read or write permission on the requested document**.

```typescript
// authenticate.ts — data.documentName is available but never used
return async (data: onAuthenticatePayload) => {
  const token = data.token || '';
  const result = await tokenVerifier.verifyToken(token);
  if (!result.ok) throw new Error(result.error.message);
  return { principal: result.principal }; // ← no permission check on data.documentName
};
```

**Impact:** Any user with a valid JWT can open a WebSocket connection to *any* document's CRDT session. This allows:
- Reading the full content of any document they are not authorized to view
- Writing arbitrary changes to any document they are not authorized to edit
- Completely bypassing the grants system for the application's primary feature

**Remediation:** In `onAuthenticate`, after verifying the token, call `permissions.checkPermission(principal.id, data.documentName, 'read')` (or `'write'` for write connections) and throw if the check fails. The `PermissionsModule` must be injected into `createCollabServer`'s dependencies.

```typescript
export function createOnAuthenticate(tokenVerifier: TokenVerifier, permissions: PermissionsModule) {
  return async (data: onAuthenticatePayload) => {
    const result = await tokenVerifier.verifyToken(data.token || '');
    if (!result.ok) throw new Error(result.error.message);
    const allowed = await permissions.checkPermission(result.principal.id, data.documentName, 'read');
    if (!allowed) throw new Error('Forbidden');
    return { principal: result.principal };
  };
}
```

## High Findings

### HIGH-1: Reference `GET /:id` — No Library Access Check (IDOR)

**File:** `modules/api/internal/reference-routes.ts`, lines 127–134  
**Severity:** HIGH

All other reference endpoints (list, create, update, delete) call `checkLibraryAccess` before acting. The `GET /:id` route applies only `requireAuth`:

```typescript
router.get('/:id', permissions.requireAuth, asyncHandler(async (req, res) => {
  const ref = await getReference(String(req.params.id));
  if (!ref) { res.status(404).json(...); return; }
  res.json(ref); // ← no library access check
}));
```

**Impact:** Any authenticated user can fetch any reference by UUID, regardless of whether they have read access to the workspace/library that owns that reference. An attacker with a valid session can enumerate or guess reference UUIDs and exfiltrate content.

**Remediation:** Add the same `checkLibraryAccess(permissions.grantStore, principal.id, workspaceId, 'read')` guard that the list endpoint uses:

```typescript
router.get('/:id', permissions.requireAuth, asyncHandler(async (req, res) => {
  const workspaceId = '00000000-0000-0000-0000-000000000000';
  const principal = req.principal!;
  const canRead = await checkLibraryAccess(permissions.grantStore, principal.id, workspaceId, 'read');
  if (!canRead) { res.status(403).json({ error: 'No read access to reference library' }); return; }
  const ref = await getReference(String(req.params.id));
  ...
}));
```

### HIGH-2: Folder Listing — No Ownership Filtering (IDOR)

**File:** `modules/api/internal/folder-routes.ts`, lines 43–51  
**Severity:** HIGH

The document listing endpoint (`GET /documents`) correctly filters results by the principal's grants. The folder listing endpoint does not:

```typescript
router.get('/', permissions.requireAuth, asyncHandler(async (req, res) => {
  const folders = await listFolders(queryResult.data.parentId ?? null);
  res.json(folders); // ← returns ALL folders to any authenticated user
}));
```

**Impact:** Any authenticated user can enumerate all folders in the system, revealing the organizational structure of other users' document trees. This leaks potentially sensitive metadata (folder names, hierarchy).

**Remediation:** Filter by grants, matching the pattern in `document-routes.ts`:

```typescript
if (loadConfig().auth.mode !== 'dev') {
  const grants = await permissions.grantStore.findByPrincipal(principal.id);
  const allowedIds = new Set(
    grants.filter((g) => g.resourceType === 'folder').map((g) => g.resourceId)
  );
  return res.json(folders.filter((f) => allowedIds.has(f.id)));
}
```

### HIGH-3: Upload Route — Auth Check After Multer (Pre-auth Memory Exhaustion)

**File:** `modules/api/internal/upload-routes.ts`, lines 55–58  
**Severity:** HIGH

The upload route applies `upload.single('file')` (multer, 10 MB limit) *before* checking authentication:

```typescript
router.post(
  '/upload',
  upload.single('file'),  // ← runs before auth
  asyncHandler(async (req, res) => {
    const principal = req.principal;
    if (!principal) { res.status(401).json(...); return; } // ← too late
```

**Impact:** An unauthenticated attacker can cause the server to buffer up to 10 MB per request in memory before the 401 is returned. Under sustained load this is an effective DoS vector against the API server process. No authentication, rate limiting on the file parse path, or proof-of-work is required.

**Remediation:** Apply `permissions.requireAuth` middleware *before* multer:

```typescript
router.post(
  '/upload',
  permissions.requireAuth,  // ← authenticate first
  upload.single('file'),    // ← then parse the upload
  asyncHandler(async (req, res) => { ... })
);
```

## Medium Findings

### MED-1: HMAC Digest Comparison Not Timing-Safe

**File:** `modules/audit/internal/hmac-chain.ts`, line 44  
**Severity:** MEDIUM

```typescript
return expected === entry.hash; // ← timing-safe comparison not used
```

String `===` comparison in JavaScript may short-circuit on the first differing character, leaking information about the correct hash via timing side-channels.

**Remediation:** Use `crypto.timingSafeEqual`:

```typescript
import { timingSafeEqual } from 'node:crypto';
const a = Buffer.from(expected, 'hex');
const b = Buffer.from(entry.hash, 'hex');
return a.length === b.length && timingSafeEqual(a, b);
```

### MED-2: `transferTo` Not Validated in Data Purge Endpoint

**File:** `modules/api/internal/admin-routes.ts`, line 57  
**Severity:** MEDIUM

```typescript
const transferTo = req.query.transferTo as string | undefined;
```

The `transferTo` value is used as a `principalId` when creating new ownership grants, but is never validated to be a well-formed UUID or a known user. A user could transfer their documents to an arbitrary string, creating orphaned grants referencing a non-existent principal.

**Remediation:** Validate with Zod (e.g., `z.string().uuid()`) and optionally verify the target principal exists:

```typescript
const transferToResult = z.string().uuid().optional().safeParse(req.query.transferTo);
if (!transferToResult.success) {
  res.status(400).json({ error: 'transferTo must be a valid UUID' });
  return;
}
const transferTo = transferToResult.data;
```

### MED-3: Rate Limit TOCTOU on Share Link Password Attempts

**File:** `modules/sharing/internal/routes.ts`, lines 64–68  
**Severity:** MEDIUM

The rate limiter's `check()` and `record()` calls are not atomic:

```typescript
const allowed = await rateLimiter.check(token);   // (1) read count
if (!allowed) { res.status(429)...; return; }
// ... attempt resolution ...
await rateLimiter.record(token);                  // (2) increment (only on failure)
```

Between (1) and (2), concurrent requests can all pass the `check()` before any `record()` runs, allowing more than `maxAttempts` password tries in the same window.

**Remediation:** Replace the check-then-record pattern with an atomic `INCR`-first approach in Redis (increment unconditionally, then check the resulting count). Consider using a Lua script or the `incr`-return-value directly:

```typescript
const count = await rateLimiter.increment(token); // atomic INCR + EXPIRE
if (count > maxAttempts) {
  res.status(429)...; return;
}
```

### MED-4: Reference Import — No Explicit Body Size Limit

**File:** `modules/api/internal/reference-import-routes.ts`  
**Severity:** MEDIUM

The import endpoint reads `req.body` as text. If the global body parser limit is not enforced (or is set generously), a large BibTeX/RIS file could exhaust server memory. The parsing itself (`parseBibTeX`, `parseRIS`) operates on the full string.

**Remediation:** Apply an explicit `express.text({ limit: '1mb' })` middleware on this route, and consider a max-entry guard after parsing (e.g., reject if `parsed.length > 500`).

### LOW-1: `(req as any)` Type Cast in Audit Routes

**File:** `modules/audit/internal/audit-routes.ts`, line 54  
**Severity:** LOW

```typescript
const principal = (req as any).principal;
```

`requireForResource` middleware already ensures `req.principal` is set. The cast bypasses TypeScript's type system unnecessarily. Use the typed accessor: `req.principal!`.

### LOW-2: DOI/ISBN Error Messages May Leak Upstream Details

**File:** `modules/references/internal/doi-lookup.ts`, lines 169–171  
**Severity:** LOW

```typescript
const msg = err instanceof Error ? err.message : String(err);
return { ok: false, error: { code: 'UPSTREAM_ERROR', message: msg } };
```

Raw Node.js/fetch error messages (e.g., DNS resolution failures, connection refused) can reveal internal network topology when returned to clients. Log the full error server-side and return a sanitized message: `'External lookup service unavailable'`.

### LOW-3: `loadConfig()` Called Per-Request in Middleware

**File:** `modules/permissions/internal/middleware.ts`, `modules/api/internal/search-routes.ts`, etc.  
**Severity:** LOW

`loadConfig()` is called inside request handlers on each invocation. While `process.env` access is fast, and the config is frozen after the first parse, repeated Zod parsing per request has nonzero overhead. This is a performance concern, not a security issue. Consider memoizing the return value at module initialization.

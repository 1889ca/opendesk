# Contract: kb

## Purpose

Manage knowledge base entries with lifecycle governance (draft/reviewed/published/deprecated) and version-pinned references for cross-document citation and RAG extraction.

## Inputs

- `entryCreate`: `KbEntryCreateInput` -- Title, body, tags, and optional metadata for a new KB entry.
- `entryUpdate`: `KbEntryUpdateInput` -- Partial update to an existing entry's content fields (creates a new version).
- `entryId`: `string (UUID)` -- Identifier for a specific KB entry.
- `workspaceId`: `string (UUID)` -- Scopes KB operations to a workspace.
- `statusTransition`: `{ entryId, from, to }` -- Validated lifecycle transition request.
- `versionRef`: `{ entryId, version: number | 'latest' }` -- Version resolution request.

## Outputs

- `KbEntry`: Full KB entry record including status and current version number.
- `KbEntryVersion`: A specific version snapshot of a KB entry's content.
- `ResolvedReference`: `{ entryId, version, title, body, resolvedAt }` -- Resolved content for a pinned or latest reference.
- `StatusTransitionResult`: `{ ok, entry, previousStatus }` -- Result of a lifecycle transition.

## Side Effects

- Persists KB entries and version snapshots to PostgreSQL via the `storage` module's pool.
- Updating an entry creates a new version row; previous versions are immutable.
- Status transitions are validated against the allowed transition graph.

## Invariants

- Entry IDs are UUIDs, generated by the caller.
- New entries always start in `draft` status.
- Allowed transitions: draft->reviewed, reviewed->published, reviewed->draft, published->deprecated.
- Only `published` entries are returned by RAG/federation/citation queries.
- Version numbers are monotonically increasing per entry, starting at 1.
- Version snapshots are immutable once created.
- `kb://entry-uuid@v7` resolves to version 7's content (or null if it does not exist).
- `kb://entry-uuid@latest` resolves to the highest-versioned content of the current published entry.
- Workspace scoping is mandatory: every query filters by `workspaceId`.

## Dependencies

- `storage` -- Provides the PostgreSQL connection pool.
- `permissions` -- Callers must check grants before invoking mutations.

## Boundary Rules

- MUST: Scope all queries by `workspaceId`.
- MUST: Validate all inputs with Zod schemas before database operations.
- MUST: Use parameterized queries for all SQL.
- MUST: Return `null` (not throw) when a requested entry does not exist.
- MUST: Create a new version row on every content update.
- MUST NOT: Import editor types (TipTap, ProseMirror).
- MUST NOT: Handle HTTP requests or routing.
- MUST NOT: Enforce permissions internally.
- MUST NOT: Manage database connections or pool lifecycle.

## Verification

- New entries default to draft -> Unit test: create entry, assert status === 'draft'.
- Status transitions are validated -> Unit test: attempt invalid transition (draft->published), assert rejection.
- Version immutability -> Integration test: update entry, verify previous version content unchanged.
- Workspace scoping -> Integration test: entries in workspace A not visible from workspace B queries.
- Pinned version resolution -> Unit test: resolve `@v3`, assert correct historical content returned.
- Latest resolution -> Unit test: resolve `@latest`, assert current published version returned.
- Only published entries for RAG -> Integration test: query published entries, assert no drafts/deprecated.

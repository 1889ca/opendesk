# Contract: kb

## Purpose

Define the canonical data types, schemas, and lifecycle for Knowledge Base entries. KB entries are typed knowledge artifacts (reference, entity, dataset, note, glossary) scoped to a workspace. Each entry has a corpus partition (knowledge, operational, reference) that controls how it feeds into RAG and other AI pipelines.

## Inputs

- `entryCreate`: `KbEntryCreateInput` -- Type-discriminated input for creating a new KB entry.
- `entryUpdate`: `KbEntryUpdateInput` -- Partial update to an existing entry's fields.
- `entryId`: `string (UUID)` -- Identifier for a specific KB entry.
- `workspaceId`: `string (UUID)` -- Scopes all operations to a workspace.

## Outputs

### KbEntry (discriminated union)

The single load-bearing type. All KB consumers switch on `entryType`.

```typescript
type KbEntryType = 'reference' | 'entity' | 'dataset' | 'note' | 'glossary';
type KbCorpus = 'knowledge' | 'operational' | 'reference';
type KbLifecycle = 'draft' | 'published' | 'archived';

type KbEntryBase = {
  id: string;                   // UUIDv4
  workspaceId: string;          // UUIDv4
  entryType: KbEntryType;
  corpus: KbCorpus;
  lifecycle: KbLifecycle;
  title: string;
  tags: string[];
  createdBy: string;
  createdAt: string;            // ISO 8601
  updatedAt: string;            // ISO 8601
};
```

### Entry type variants

- **ReferenceEntry**: `{ entryType: 'reference', authors: Author[], abstract: string | null, metadata: Record<string, unknown> }`
- **EntityEntry**: `{ entryType: 'entity', description: string, metadata: Record<string, unknown> }`
- **DatasetEntry**: `{ entryType: 'dataset', description: string, columns: DatasetColumn[], summary: string | null }`
- **NoteEntry**: `{ entryType: 'note', content: string }`
- **GlossaryEntry**: `{ entryType: 'glossary', term: string, definition: string }`

## Side Effects

- Persists entries to PostgreSQL via the `storage` module's pool.

## Invariants

1. **Entry IDs are UUIDs.** Generated by the caller.
2. **Workspace scoping is mandatory.** Every query filters by `workspaceId`.
3. **Lifecycle transitions are ordered.** draft -> published -> archived. No skipping, no backward.
4. **Corpus partition is immutable after creation.** Set once at creation time.
5. **Entry type is immutable.** Cannot change after creation.
6. **Title is non-empty.** Every entry has a title.

## Dependencies

- `storage` -- Provides the PostgreSQL connection pool.
- `references` -- Shares the `Author` type for reference entries.

## Boundary Rules

- MUST: Scope all queries by `workspaceId`.
- MUST: Validate all inputs with Zod schemas before database operations.
- MUST: Use parameterized queries for all SQL.
- MUST: Return `null` (not throw) when a requested entry does not exist.
- MUST NOT: Handle HTTP requests, routing, or middleware.
- MUST NOT: Enforce permissions internally. Callers are responsible.
- MUST NOT: Manage database connections or pool lifecycle.
- MUST NOT: Perform embedding or AI operations. That belongs to the `ai` module.

## Verification

- Entry ID format -> Unit test: parse every created entry's `id` with UUID regex.
- Workspace scoping -> Integration test: entries in workspace A not visible from workspace B.
- Lifecycle transitions -> Unit test: verify only valid transitions are accepted.
- Corpus immutability -> Unit test: verify update rejects corpus field changes.
- Entry type immutability -> Unit test: verify update rejects entryType changes.
- Title non-empty -> Unit test: verify schema rejects empty title.
